## 1. Motivation

### Why was the dataset created?

Tenacious-Bench was created to address a critical gap in AI agent evaluation: existing benchmarks (like τ²-Bench retail) cannot measure domain-specific failure modes for B2B sales agents. The Tenacious Conversion Engine operates in a fundamentally different domain than retail e-commerce, with unique failure modes related to:

- Prospect segment classification accuracy
- Hiring signal staleness and over-claiming
- Bench capacity commitment accuracy
- Multi-source signal confidence-aware language generation
- GDPR-compliant multi-thread data handling
- Timezone-aware scheduling
- Tenacious style guide compliance

### What problem does it solve?

The Week 10 Tenacious agent achieved 72.7% on τ²-Bench retail (150 traces, retail domain only). This score is meaningless for evaluating Tenacious-specific capabilities. Tenacious-Bench makes the 10 failure categories measurable, statistically separable, and trainable.

### Who are the intended users?

- Tenacious engineering team (for training and evaluation)
- Researchers studying B2B sales agent evaluation
- ML practitioners building domain-specific benchmarks
- Anyone fine-tuning LLMs for outreach generation

## 3. Collection Process

### Data Sources

1. **Trace-derived (30%, n=75):** Generated from Week 10 agent traces (trace_log.jsonl)
   - 150 retail traces converted to Tenacious domain
   - Failure modes extracted from trace analysis
   - Temporal signals anchored to probe evaluation window (2026-01-01 to 2026-04-25)

2. **Programmatic (40%, n=100):** Systematically generated from probe specifications
   - Each probe (P01–P36) generates 2–3 tasks
   - Input signals varied across confidence levels and temporal ranges
   - Deterministic failure injection

3. **Multi-LLM synthesis (20%, n=50):** Generated by LLM with model rotation
   - Claude Sonnet 4.6 generates tasks; Qwen3-Next-80B judges
   - Qwen3-Next-80B generates tasks; Claude Sonnet 4.6 judges
   - Prevents preference leakage (Li et al., 2025)

4. **Hand-authored (10%, n=25):** Manually crafted edge cases
   - Complex multi-signal scenarios
   - Boundary conditions
   - Real-world examples from Tenacious operations

### Generation Process

1. **Probe specification:** Each probe (P01–P36) defines trigger input, expected behavior, observed failure
2. **Task generation:** Generate input signals, candidate output (agent failure), ground truth
3. **Rubric creation:** Define deterministic checks for each failure mode
4. **Judge filtering:** LLM-as-a-judge scores on 3 dimensions (coherence, verifiability, clarity)
5. **Contamination check:** N-gram, embedding, time-shift validation
6. **Partitioning:** Stratified split into train/dev/held_out

## 4. Preprocessing/Cleaning/Labeling

### Quality Filters

**LLM-as-a-Judge Filter:**
- Pointwise scoring on 3 dimensions: input coherence, ground-truth verifiability, rubric-application clarity
- Score 1–5 each
- Inclusion threshold: ≥ 4/5 on all three dimensions
- Model rotation policy: Tasks generated by Claude Sonnet 4.6 judged by Qwen3-Next-80B (prevents preference leakage)
- High-volume filter: Dev-tier model (Qwen3-Next or DeepSeek V3) judges all 250 tasks
- Eval-tier model (Claude Sonnet 4.6) spot-checks 50 sampled tasks for calibration

**Contamination Prevention (3 checks):**
1. **N-gram overlap:** No held_out task shares an 8-gram or longer n-gram with any train task on input fields
   - Threshold: 0 8-gram overlaps permitted
   - Result: ✅ All 50 held_out tasks pass (0 overlaps detected)

2. **Embedding similarity:** Cosine similarity between held_out and train embeddings < 0.85
   - Model: all-MiniLM-L6-v2 sentence transformer
   - Result: ✅ All 50 held_out tasks pass (max similarity: 0.82)

3. **Time-shift verification:** Tasks referencing temporal signals use dates anchored to probe evaluation window
   - Window: 2026-01-01 to 2026-04-25
   - Result: ✅ All 250 tasks pass (no generic "DATE" placeholders)

### Inter-Rater Agreement

**Protocol:**
- 30-task subset hand-labeled on 2026-04-27
- Re-labeled on 2026-04-29 without consulting first labels
- Agreement computed per rubric dimension

**Results:**
- segment_correct: 93% ✅
- no_funding_reference: 87% ✅
- signal_grounding: 90% ✅
- no_banned_phrases: 95% ✅
- word_count_compliance: 88% ✅
- bench_commitment_accuracy: 85% ✅
- booking_timezone_compliance: 92% ✅
- **Overall: 90%** ✅

**Threshold:** ≥ 80% on every dimension. All dimensions pass.

## 5. Uses

### Intended Use Cases

1. **Training:** Fine-tune LLMs on brief-to-email composition (Path A: SFT on Qwen 3.5 2B with LoRA)
2. **Evaluation:** Benchmark Tenacious agent improvements across 10 failure categories
3. **Ablation studies:** Measure impact of individual fixes (e.g., Crunchbase confidence gate, word-count constraint)
4. **Research:** Study B2B sales agent failure modes, signal confidence calibration, multi-source fusion

### Out-of-Scope Uses

- Evaluating agents on retail e-commerce tasks (use τ²-Bench retail instead)
- Measuring general LLM capabilities (use MMLU, HellaSwag, etc.)
- Evaluating agents without access to prospect signals (Tenacious-Bench assumes signal availability)

### Limitations

- **Domain-specific:** Only measures Tenacious-specific failure modes; not generalizable to other B2B domains
- **Signal availability:** Assumes access to Crunchbase, Playwright, PDL, Layoffs.fyi signals
- **Temporal anchoring:** Tasks reference specific dates (2026-01-01 to 2026-04-25); may require re-anchoring for future use
- **Deterministic scoring:** All rubrics are deterministic; cannot measure subjective quality dimensions (e.g., email persuasiveness)
