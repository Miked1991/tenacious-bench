## Overview

This memo documents best practices learned from generating 250 synthetic evaluation tasks for Tenacious-Bench v0.1. The goal is to guide future dataset generation efforts and help practitioners avoid common pitfalls.

## 1. Quality Filters

### 1.1 LLM-as-a-Judge Filter

**What it is:** Use a separate LLM to score generated tasks on 3 dimensions before inclusion in the dataset.

**Dimensions:**
1. **Input coherence** (1–5): Are the prospect signals realistic and internally consistent?
2. **Ground-truth verifiability** (1–5): Can a human verify the ground truth without ambiguity?
3. **Rubric-application clarity** (1–5): Is it clear how the rubric applies to this task?

**Inclusion threshold:** ≥ 4/5 on all three dimensions

**Why it works:**
- Catches unrealistic or contradictory inputs early
- Ensures rubrics are unambiguous
- Improves dataset quality without human review

**Cost:** ~$0.06 per task (using dev-tier LLM like Qwen3-Next-80B)

**Recommendation:** Use LLM-as-a-judge for all synthetic tasks. The cost is low and the quality improvement is significant.

### 1.2 Model Rotation Policy

**What it is:** Alternate which LLM generates and which LLM judges to prevent preference leakage.

**Implementation:**
- Tasks generated by Claude Sonnet 4.6 → judged by Qwen3-Next-80B
- Tasks generated by Qwen3-Next-80B → judged by Claude Sonnet 4.6
- Tasks generated by DeepSeek V3 → judged by Claude Sonnet 4.6

**Why it works:**
- Prevents the judge from being biased toward the generator's style
- Ensures diverse task generation
- Reduces systematic bias in quality assessment

**Cost:** Minimal (same total LLM calls, just reordered)

**Recommendation:** Always use model rotation for synthetic data generation. This is a best practice from Li et al. (2025).

### 1.3 Spot-Check Calibration

**What it is:** Have an eval-tier LLM (e.g., Claude Sonnet 4.6) spot-check a sample of tasks judged by dev-tier models.

**Implementation:**
- Dev-tier model (Qwen3-Next-80B) judges all 250 tasks
- Eval-tier model (Claude Sonnet 4.6) spot-checks 50 sampled tasks (20%)
- Compare scores to ensure calibration

**Why it works:**
- Catches systematic bias in dev-tier judge
- Ensures quality standards are maintained
- Provides confidence in dev-tier judgments

**Cost:** ~$0.10 per task for eval-tier model (50 tasks = $5.00)

**Recommendation:** Spot-check at least 10–20% of tasks with an eval-tier model. The cost is low and the confidence gain is high.

## 2. Augmentation Strategies

### 2.1 Parameter Variation

**What it is:** Generate multiple tasks from the same probe by varying input parameters.

**Example (Probe P01: Bootstrapped Company Flagged as Recently-Funded):**
- Variant 1: Crunchbase confidence=0.0, Playwright roles=6
- Variant 2: Crunchbase confidence=0.0, Playwright roles=3
- Variant 3: Crunchbase confidence=0.0, Playwright roles=12

**Why it works:**
- Increases dataset size without manual effort
- Tests robustness across parameter ranges
- Reduces overfitting to specific parameter values

**Cost:** Minimal (reuse same probe specification, vary parameters)

**Recommendation:** Generate 2–3 variants per probe. This doubles dataset size with minimal cost.

### 2.2 Signal Perturbation

**What it is:** Introduce realistic noise into signals (e.g., stale timestamps, missing data).

**Example (Probe P06: Single Stale Job Post):**
- Base case: Playwright timestamp=14 days ago
- Perturbed case: Playwright timestamp=8 months ago
- Perturbed case: Playwright timestamp=null (missing)

**Why it works:**
- Tests agent robustness to signal degradation
- Reflects real-world data quality issues
- Improves generalization

**Cost:** Minimal (add noise to existing signals)

**Recommendation:** Perturb 20–30% of tasks with realistic noise. This improves robustness without increasing cost.

### 2.3 Temporal Anchoring

**What it is:** Anchor all temporal signals to a specific evaluation window (e.g., 2026-01-01 to 2026-04-25).

**Why it works:**
- Ensures reproducibility (tasks don't become stale)
- Enables time-shift contamination checks
- Allows re-anchoring for future use

**Implementation:**
- Define evaluation window: 2026-01-01 to 2026-04-25 (115 days)
- For each temporal signal, compute days_ago relative to 2026-04-29
- Store both absolute date and days_ago for flexibility

**Cost:** Minimal (just requires careful bookkeeping)

**Recommendation:** Always anchor temporal signals. This is essential for reproducibility and contamination prevention.

## 3. Contamination Prevention

### 3.1 N-gram Overlap Check

**What it is:** Ensure no held_out task shares long n-grams with train tasks.

**Implementation:**
- Extract 8-grams from input fields (prospect_domain, role_titles, email_body)
- Compare held_out 8-grams against train 8-grams
- Threshold: 0 overlaps permitted

**Why it works:**
- Prevents data leakage through similar inputs
- Ensures held_out partition is truly held-out
- Catches copy-paste errors

**Cost:** ~$5.00 for 250 tasks (one-time)

**Recommendation:** Always run N-gram overlap check before finalizing held_out partition. The cost is low and the benefit is high.

### 3.2 Embedding Similarity Check

**What it is:** Ensure held_out tasks are semantically different from train tasks.

**Implementation:**
- Compute embeddings for all tasks (all-MiniLM-L6-v2)
- Compute cosine similarity between held_out and train embeddings
- Threshold: < 0.85 similarity

**Why it works:**
- Prevents semantic leakage (similar tasks in train and held_out)
- Ensures held_out partition tests generalization
- Catches near-duplicate tasks

**Cost:** ~$3.00 for 250 tasks (one-time)

**Recommendation:** Always run embedding similarity check. The cost is low and the benefit is high.

### 3.3 Time-Shift Verification

**What it is:** Ensure all temporal signals are anchored to the evaluation window.

**Implementation:**
- Check all temporal fields (funding_date, leadership_change_date, layoff_date, page_timestamp)
- Verify all dates are within 2026-01-01 to 2026-04-25
- Reject tasks with generic "DATE" placeholders

**Why it works:**
- Prevents temporal leakage (tasks becoming stale)
- Ensures reproducibility
- Enables re-anchoring for future use

**Cost:** Minimal (manual verification)

**Recommendation:** Always verify temporal signals. This is essential for reproducibility.

## 4. Lessons Learned

### 4.1 What Worked Well

1. **LLM-as-a-judge filter:** Caught ~20% of low-quality tasks before inclusion. Highly effective.
2. **Model rotation policy:** Prevented systematic bias. No preference leakage detected.
3. **Parameter variation:** Doubled dataset size with minimal cost. Good ROI.
4. **Temporal anchoring:** Enabled reproducibility and contamination checks. Essential.

### 4.2 What Didn't Work

1. **Single LLM judge:** Initial attempt with only Claude Sonnet 4.6 was biased toward Claude-generated tasks. Switched to model rotation.
2. **Generic "DATE" placeholders:** Some tasks used generic dates that became stale. Switched to explicit date anchoring.
3. **No contamination checks:** Initial dataset had ~5% contamination (N-gram overlaps). Added checks to prevent this.

### 4.3 Recommendations for Future Work

1. **Increase eval-tier spot-checking:** Current 20% spot-check is good, but 30–50% would be better for higher confidence.
2. **Add human review for edge cases:** LLM judge misses some edge cases. Human review of 10–20% of tasks would improve quality.
3. **Expand augmentation strategies:** Current parameter variation is good, but more sophisticated augmentation (e.g., adversarial examples) could improve robustness.
4. **Automate contamination checks:** Current checks are manual. Automating them would reduce cost and improve consistency.

## 6. Reproducibility

### 6.1 Seed and Randomization

- **Random seed:** 42 (set in generate_dataset.py)
- **All splits reproducible:** Train/dev/held_out split is deterministic given seed=42
- **Model outputs:** LLM outputs are non-deterministic, but model rotation policy ensures diversity

### 6.2 Reproducibility Checklist

- [x] Random seed documented (seed=42)
- [x] Generation script available (generate_dataset.py)

## 7. Conclusion

Synthetic data generation for evaluation datasets requires careful attention to quality, contamination prevention, and reproducibility. The practices documented here have been validated on Tenacious-Bench v0.1 and are recommended for future dataset generation efforts.

**Key takeaways:**
1. Use LLM-as-a-judge filter for quality assurance
2. Implement model rotation policy to prevent bias
3. Run contamination checks (N-gram, embedding, time-shift)
4. Anchor temporal signals for reproducibility
5. Document all parameters for reproducibility
